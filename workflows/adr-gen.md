# ADR Validation Workflow

Validates completeness of Architecture Decision Records. Checks required sections, back-references, and review triggers. Creates stub ADRs for undocumented decisions.

---

## Step 0: Workspace Resolution

@~/.claude/sdlc/workflows/workspace-resolution.md

After resolution:
```bash
PHASE_ARTIFACTS="$ARTIFACTS/design"
mkdir -p "$PHASE_ARTIFACTS"
```

Note: ADR artifacts live alongside design artifacts since they are part of the design phase output.

---

## Step 1: Read Design Artifacts

Read in parallel:
- `$ARTIFACTS/design/solution-design.md` — primary ADR source
- `$ARTIFACTS/design/tech-architecture.md` — architecture decisions that may need ADRs

Also glob for any standalone ADR files:
```bash
find "$ARTIFACTS/design" -name "adr-*.md" 2>/dev/null
```

If solution-design.md does not exist:
- If `--auto-chain`: log `{ "status": "skipped-condition-not-met", "summary": "solution-design.md not found" }` and output `⏭️ adr-gen — skipped: solution-design.md not found`
- If interactive: inform the user that design phase has not produced a solution-design.md and ask whether to proceed with tech-architecture.md alone.

---

## Step 2: Extract ADR Inventory

Parse all ADR-NNN entries from the documents read in Step 1. For each ADR found, record:

```
ADR-NNN: [Title]
Source file: [which file it was found in]
Sections present:
  - Context:               [yes/no]
  - Decision:              [yes/no]
  - Alternatives Considered: [yes/no]
  - Rationale:             [yes/no]
  - Consequences:          [yes/no]
  - Review Trigger:        [yes/no — and if yes, is it specific?]
Back-referenced from tech-architecture.md: [yes/no]
```

A "specific" review trigger is one that names a measurable condition (e.g., "if monthly active users exceed 50k", "if p95 latency exceeds 500ms for 3 consecutive days"). Vague triggers like "when things change" or "revisit periodically" are NOT specific.

---

## Step 3: Validate Each ADR

For each ADR from Step 2:

### 3a: Missing sections
If any required section is absent: add it with a placeholder prompt inside the existing ADR.

Placeholder format:
```markdown
### [Missing Section Name]
> TODO: [specific prompt for what to fill in here]
> Example: "What alternatives to [decision] were considered and why were they rejected?"
```

Write the updated file back.

### 3b: Vague review trigger
If review trigger exists but is vague: flag it and suggest a specific version.

Flag format: `⚠️ ADR-NNN review trigger is vague: "[current text]" — suggest: "[specific measurable version]"`

### 3c: Missing back-reference
If the ADR is not referenced from tech-architecture.md: note it for reporting but do NOT auto-edit tech-architecture.md (too risky to modify that file silently).

Flag format: `📎 ADR-NNN not referenced in tech-architecture.md — add reference manually`

---

## Step 4: Scan for Undocumented Decisions

Read tech-architecture.md and look for major architectural decisions that do NOT have a corresponding ADR. Look for:

- Technology choices (database, framework, message broker, cache, auth provider)
- Deployment topology (monolith / microservices / serverless)
- API versioning strategy
- Authentication and authorization model
- Data encryption approach
- Event-driven vs synchronous communication choices
- Caching strategy
- External service selections

Cross-reference against the ADR inventory from Step 2. For each decision found in the architecture with no ADR coverage:

Create a stub ADR file at `$PHASE_ARTIFACTS/adr-NNN-[kebab-title].md`:

```markdown
## ADR-NNN: [Decision Title — derived from the undocumented decision]
*Date: [today] | Status: Draft — stub generated by adr-gen*

### Context
> TODO: Describe the situation and constraints that led to this decision.

### Decision
[Derived from tech-architecture.md — state what was decided]

### Alternatives Considered
> TODO: List alternatives that were evaluated and why each was rejected.

### Rationale
> TODO: Explain why this option was chosen over the alternatives given the constraints.

### Consequences
- Positive: > TODO
- Negative: > TODO
- Neutral: > TODO

### Review Trigger
> TODO: Define a specific, measurable condition under which this decision should be revisited.
> Example: "If [metric] exceeds [threshold] for [duration]"
```

Number stubs starting after the highest existing ADR number.

---

## Step 5: Write Summary Report

Append or create `$PHASE_ARTIFACTS/adr-validation-report.md`:

```markdown
# ADR Validation Report
*Generated: [date]*

## Summary
- ADRs found: [N]
- ADRs fully complete: [N]
- ADRs with missing sections: [N] — sections added with placeholders
- ADRs with vague review triggers: [N]
- ADRs missing back-reference in tech-architecture.md: [N]
- Stub ADRs created for undocumented decisions: [N]

## Validation Results

| ADR | Title | Complete | Review Trigger | Back-Referenced | Issues |
|-----|-------|----------|----------------|-----------------|--------|
[one row per ADR]

## Flags Requiring Manual Attention
[List of ⚠️ and 📎 flags from Steps 3b and 3c]

## New Stub ADRs Created
[List of file paths created in Step 4]
```

---

## Step 5b: Test Coverage Check per ADR

For each ADR that has a "Review Trigger" with test implications (look for language like "if performance degrades", "if security incident occurs", "if [metric] exceeds"), check whether test-cases.md covers the ADR's key commitment.

Read `$ARTIFACTS/test-cases/test-cases.md` (if exists). For each ADR:
  - Extract the core architectural commitment (the "Decision" section)
  - Search test-cases.md for TC-IDs that reference this ADR or test the behavior it commits to
  - If no TC-ID found: flag as a coverage gap

Append to `$PHASE_ARTIFACTS/adr-validation-report.md`:

```markdown
## ADR Test Coverage

| ADR | Decision Summary | Covering TC-IDs | Coverage Status |
|-----|-----------------|-----------------|----------------|
| ADR-001 | [one-line decision] | TC-023, TC-024 | ✅ Covered |
| ADR-002 | [one-line decision] | — | ❌ No test coverage |
```

**Gap actions:**
For each ADR with no test coverage:
- Add a flag: `⚠️ ADR-NNN has no test coverage — add TC-ID that validates "[decision commitment]"`
- If test-cases.md exists: append a stub TC-ID entry:
  ```
  TC-ADR-NNN: Validate ADR-NNN — [decision title]
  Layer: [most appropriate layer — Unit/Integration/Performance/Security]
  Note: Stub generated by adr-test-coverage. Fill in Given/When/Then from ADR decision.
  Given: The system uses [ADR decision]
  When: [scenario that tests the decision holds]
  Then: [expected outcome that would fail if ADR was violated]
  ```

Include ADR test coverage counts in the Step 6 state.json summary.

